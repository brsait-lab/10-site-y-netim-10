## Architecture Decisions

### Unit Key Pattern
`buildUnitKey(user)` → `"A-12"` (block+unit), `"K1-3"` (tower+unit), `"villa-4"` (villaNo), `"12"` (unitNo only), `null` if none.
Unit-based payments create one UserPayment per unique unitKey (not per user). All residents in same unit see the same record.

### isUnitBased(type)
Returns true for: `aidat`, `extra_expense`, `gider` (legacy alias). False for `personal_charge`.

### Status Flow
`pending` → (resident uploads dekont) → `pending_approval` → (admin approves/rejects) → `paid` / `rejected`
Admin can also do `manual-pay` (cash/bank_transfer) directly → `paid`

### New Prisma Models
- `PaymentAuditLog`: tracks all payment actions with performedByName (looked up from DB at write time)
- `Expense`: site-level expense records (gider kalemi), separate from Payment

### Payment Model Additions
Added: `year`, `month`, `period` (derived from dueDate on create), `createdBy`, `cancelledAt`

### UserPayment Model Additions  
Added: `unitKey` (nullable), `userId` (now nullable), `paidByUserId`, `paymentMethod`, `approvedBy`, `approvedAt`, `receiptUrl`

### Security Model
- `blockSecurity = blockRoles("merchant","security")` — security blocked from ALL payment/expense endpoints
- `blockNonAdmin = blockRoles("merchant","resident","security")` — admin-only actions

### Sites.ts Resident IBAN Access
GET /sites/:id now returns `bankName`, `accountHolder`, `iban` (decrypted) to residents of that site (for payment UX). No `joinCode`.

### New API Endpoints
- `PATCH /user-payments/:id/upload-receipt` — resident submits dekont URL
- `PATCH /user-payments/:id/approve` — admin approves pending_approval
- `PATCH /user-payments/:id/reject` — admin rejects pending_approval  
- `PATCH /user-payments/:id/manual-pay` — admin records cash/manual payment
- `GET /payment-audit-logs` — admin views audit history
- `GET /expenses`, `POST /expenses`, `DELETE /expenses/:id` — expense CRUD

### Client Functions Location
New functions in `lib/api-client-react/src/generated/payments-extra.ts`, exported from `src/index.ts`.
Do NOT add to generated `api.ts` (too large, regenerated by orval).

### DataContext New Exports
`expenses: Expense[]`, `pendingApprovalCount: number`, plus all new action functions.
Expenses not loaded for security role.

**Why:** Separation of generated code (api.ts via orval) from hand-written extras (payments-extra.ts) prevents regeneration from wiping new functions.
